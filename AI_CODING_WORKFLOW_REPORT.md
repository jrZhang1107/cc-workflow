## 一、背景：AI Coding 落地的五大痛点

在企业级软件开发实践中，AI Agent 虽展现巨大潜力，但仍面临以下核心挑战：

| 痛点 | 问题描述 | 影响 |
|------|---------|------|
| **AI 不懂业务** | 每次任务前需阅读大量源码理解项目 | 启动成本高，效率低下 |
| **多轮对话"变笨"** | Agent 通过 find/grep/cat 阅读源码，上下文信息密度低 | 多轮后模型降智，输出质量下降 |
| **任务串行执行** | 单线程交互无法同时处理多个独立模块 | 复杂需求下时间成本线性增长 |
| **工具/模型单一** | 单一工具存在短板，无法获取最优效果 | 不同场景无法发挥各模型优势 |
| **缺少团队资产沉淀** | Prompt、Skills 停留在个人经验层面 | 个人智慧无法转化为团队资产 |

### 当前低效的工作模式

```
用户 → 单一 Agent → grep/find/cat (低密度信息) → 串行理解 → 完成任务
                          ↑
                    上下文窗口被污染
                    多轮后性能下降
```

## 二、解决方案：cc-workflow 标准化工作流

针对上述痛点，我搭建了 **cc-workflow** 框架，提供三大核心能力：

### 2.1 llmdoc 文档驱动开发

**解决痛点：AI 不懂业务 + 多轮对话变笨**

llmdoc 是一套 LLM 友好的项目文档系统，让 Agent 快速理解项目且保持高信息密度。

#### 文档结构

```
llmdoc/
├── index.md              # 导航入口
├── overview/             # "这是什么项目？" - 高层上下文
├── architecture/         # "它是如何工作的？" - LLM 检索地图
├── guides/               # "我该如何做 X？" - 分步工作流
└── reference/            # "具体细节是什么？" - 规范与配置
```

#### 核心设计原则

| 原则 | 说明 |
|------|------|
| **LLM 优先** | 文档是为 LLM 准备的检索地图，非人类阅读书籍 |
| **代码引用** | 不粘贴代码块，使用 `path/file.ext:line` 格式引用 |
| **简洁性** | 每个文档不超过 150 行，复杂主题拆分多文件 |
| **自动维护** | 编码完成后自动触发文档更新，保持同步 |

#### 自动化初始化流程

通过 `/cc-workflow:initDoc` 命令一键生成：

```
1. Scout 代理并行探索代码库 (最多4个)
     ↓
2. 综合分析，提取核心概念
     ↓
3. 用户选择需要文档化的概念
     ↓
4. Recorder 代理并行生成文档
     ↓
5. 构建最终索引
```

### 2.2 SubAgent RAG 高密度信息投喂

**解决痛点：多轮对话变笨 + 缺少团队资产沉淀**

将信息检索任务交给专职 SubAgent，实现"高密度信息"精准投喂。

#### Agent 角色分工

| Agent | 职责 | 特点 |
|-------|------|------|
| **Investigator** | 快速代码库分析 | 文档优先，无状态，输出精简报告 |
| **Scout** | 深度探索特定模块 | 并行执行，生成临时报告 |
| **Recorder** | 创建/维护 llmdoc | 4类别结构，代码引用策略 |
| **Worker** | 执行具体编码任务 | 基于文档上下文工作 |

#### 信息流闭环

```
任务开始 → Investigator 检索 llmdoc → 高密度上下文投喂
                                            ↓
                                      Worker 执行任务
                                            ↓
任务完成 ← Recorder 更新 llmdoc ← 自动触发记录
```

**关键优势**：
- 主 Agent 上下文窗口保持干净
- 检索结果高度结构化
- 知识自动沉淀，形成闭环

### 2.3 多 CLI 协作并行执行

**解决痛点：任务串行执行 + 工具/模型单一**

Claude Code 负责任务编排，将不同任务分发给不同 CLI 并行执行。

#### 四阶段工作流

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Research   │ ──► │    Plan     │ ──► │    Exec     │ ──► │   Review    │
│  (并行分析)  │     │  (任务规划)  │     │  (并行执行)  │     │  (交叉审查)  │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
```

| 阶段 | 目的 | 工具分配 | 输出 |
|------|------|----------|------|
| **Research** | 并行分析代码库 | Codex(后端) + Gemini(前端) | 约束集、依赖关系 |
| **Plan** | 任务拆分与依赖规划 | Gemini | 任务列表 + 执行层 |
| **Exec** | 分层并行执行 | 按任务分配 | 代码变更 |
| **Review** | 双模型交叉审查 | Codex + Gemini | 问题报告 |

#### 任务分配策略

| 领域 | 推荐工具 | 原因 |
|------|---------|------|
| 后端逻辑、API、数据模型 | **Codex** | 擅长复杂逻辑和算法 |
| 前端 UI、组件、样式 | **Gemini** | 擅长设计和用户体验 |
| 复杂重构、架构调整 | **Claude** | 强大长上下文理解能力 |

#### 并行执行机制

**任务图 (TaskGraph)** 管理依赖关系，通过拓扑排序实现分层并行：

```
原始任务：
  Task A (无依赖)
  Task B (无依赖)
  Task C (依赖 A, B)
  Task D (依赖 C)

拓扑排序结果：
  Layer 1: [A, B]     ← 并行执行
  Layer 2: [C]        ← 等待 Layer 1
  Layer 3: [D]        ← 等待 Layer 2
```

**文件范围隔离** 确保并行任务不冲突：

```
✅ 可并行（无重叠）：
  Task A: src/api/**/*.js
  Task B: src/components/**/*.jsx

❌ 不可并行（有重叠）：
  Task A: src/**/*.js
  Task B: src/api/user.js  ← 自动设为依赖
```

## 四、使用方式

### 一键安装

```bash
npx cc-workflow
```

### 核心命令

| 命令 | 功能 |
|------|------|
| `/cc-workflow:initDoc` | 初始化 llmdoc 文档系统 |
| `/cc-workflow:team-research` | 并行分析代码库 |
| `/cc-workflow:team-plan` | 生成执行计划 |
| `/cc-workflow:team-exec` | 并行执行任务 |
| `/cc-workflow:team-review` | 交叉审查结果 |

### CLI 工具

```bash
# 完整工作流
llmdoc-ccw team -p "实现用户认证模块"

# 单独阶段
llmdoc-ccw team -p "分析项目" --phase research
llmdoc-ccw team --plan latest --phase exec

# 单 CLI 调用
llmdoc-ccw cli -p "分析认证模块" --tool gemini
llmdoc-ccw cli -p "实现登录功能" --tool codex --mode write
```

## 五、效果对比

### Before vs After

| 维度 | 传统方式 | cc-workflow |
|------|---------|-------------|
| **上下文理解** | 每次重新阅读源码 | llmdoc 秒级理解 |
| **信息密度** | grep/cat 低密度 | 结构化高密度投喂 |

| **知识沉淀** | 个人经验 | 自动化团队资产 |

### 预期收益

1. **效率提升**：并行执行 + 高密度上下文 → 复杂任务耗时减少 50%+
2. **质量提升**：专业模型分工 + 交叉审查 → 代码质量更高
3. **知识复用**：llmdoc 自动维护 → 团队资产持续积累
4. **标准化**：Skills + Commands → 可复制的最佳实践
